# DoD-1: стабильный и плановый автопостинг X

## Цель

Production-развёртывание надёжно выполняет запланированные публикации только
для явно одобренных материалов, а сбои Chromium диагностируются и не приводят
к ложному статусу успешной публикации.

## Критерии готовности

### 1. Устойчивость браузера в Docker

- Для сервисов, которые запускают Playwright (`api` и отдельный worker/bot,
  если он используется), в Compose настроена достаточная shared memory
  (`shm_size`, рекомендуемое значение — `1gb`) или задокументирована и
  протестирована эквивалентная безопасная конфигурация Chromium.
- Изменение применено и к local-, и к production-Compose конфигурации там,
  где сервис наследует или переопределяет параметры.
- При `Page crashed` или `Target crashed` запуск публикации получает статус
  `failed`, не помечает элемент очереди как `posted` и сохраняет понятную
  диагностическую причину в run/funnel events.
- Снимок экрана и trace сохраняются при возможности; вторичная ошибка при их
  сохранении не скрывает исходную причину сбоя браузера.
- В логах достаточно контекста для расследования: run ID, queue item ID,
  этап (warm-up/compose/post-confirmation), тип ошибки и путь к артефактам.

### 2. Планировщик публикаций

- В приложении есть явный production worker/service для планирования; он не
  зависит от HTTP-запроса из UI и запускается Compose-конфигурацией.
- Worker периодически выбирает только элементы со всеми условиями:
  - `status == approved`;
  - `scheduledFor <= now` (в UTC);
  - существует успешный dry run после последнего изменения черновика;
  - публикация разрешена настройками `DRY_RUN=false` и
    `POSTING_ENABLED=true`.
- Планировщик не публикует элементы без `scheduledFor` и не изменяет их статус.
- Применяется `MIN_POST_INTERVAL_MINUTES`: между двумя фактическими попытками
  публикации выдерживается заданный интервал.
- Нет конкурентных публикаций: один элемент не запускается дважды, а два
  worker-процесса не могут одновременно захватить один и тот же элемент.
- Повторный запуск worker после рестарта идемпотентен: уже опубликованный
  элемент не публикуется повторно.
- Время интерпретируется и сохраняется timezone-aware в UTC; поведение на
  просроченном `scheduledFor` явно определено и покрыто тестом.
- Для каждой попытки создаётся run с корректным переходом
  `queued -> running -> completed|failed|blocked` и funnel event.
- Ошибки публикации имеют ограниченный retry/backoff; повторная попытка не
  обходит approval, dry-run, интервал или флаги безопасности.

### 3. Безопасность и UX

- При `DRY_RUN=true` либо `POSTING_ENABLED=false` worker не открывает браузер
  и не отправляет запросов на X; причина фиксируется как `blocked` либо в
  структурированном логе.
- Существующая ручная публикация сохраняет требования: approve, успешный
  dry run и явное confirm из API/UI.
- В UI/API видны причина блокировки, статус последнего запуска и время
  следующей допустимой попытки, если она ограничена интервалом.
- `backend/auth.json`, `.env`, содержимое постов и session cookies не попадают
  в логи, тестовые фикстуры или commit.

### 4. Наблюдаемость

- Loguru настраивается один раз на lifecycle приложения, а не на каждом
  FastAPI dependency-вызове; одинаковые сообщения в `bot.log` не дублируются.
- Есть структурированные логи lifecycle worker: старт, поиск due items,
  захват, блокировка, запуск, результат и исключение.
- В документации указаны команды просмотра production worker, runs, logs и
  Playwright-артефактов.

### 5. Автоматическая проверка

- Unit-тесты покрывают eligibility scheduler: будущая дата, просроченная дата,
  отсутствие approval, отсутствие актуального dry run, safe flags и minimum
  interval.
- Тесты покрывают идемпотентность/защиту от двойного захвата и обработку
  browser-crash без ложного `posted`.
- Тесты логирования подтверждают отсутствие повторной установки sinks при
  нескольких API-запросах.
- Выполняются без реального обращения к X: `python -m compileall src`, тесты
  backend, `npm run build` и `docker compose ... config`.
- После деплоя выполнена только безопасная smoke-проверка при
  `DRY_RUN=true` или `POSTING_ENABLED=false`; реальная публикация возможна
  только отдельным явным решением владельца аккаунта.
